Building PyFuncAI: Prompt-Defined Functions as a Python Library

Published Mar 13, 2026

All Posts Home

Why I Built It

Most AI tooling in software today lives outside the codebase. You prompt a chatbot, copy the output, and then manually decide what belongs in the actual program. I wanted to explore a different idea: what if prompt-defined behavior could be treated as a library primitive instead of an external workflow?

That question turned into PyFuncAI, a Python library that lets you describe a function in plain language, generate the implementation through an LLM, and get back a normal callable function object.

The Core Idea

The part that interested me was not just code generation by itself. It was making code generation feel like part of the normal programming model. Instead of treating the model as an assistant sitting beside the code, PyFuncAI treats it as part of the function creation path.

A simple example looks like this:

greet = create_function("Return a short greeting for the provided name.", signature="(name: str) -> str")

The result is not a string response or a code snippet in a chat window. It is a callable Python function that can be invoked, cached, and inspected like any other runtime object.

What I Wanted The Library To Do Well

If this idea was going to be more than a novelty, it had to behave like a real library. That meant thinking about the full lifecycle around generated code, not just the first successful demo.

The implementation focuses on:

  • Lazy or eager function generation
  • Support for local and hosted models
  • Disk caching for repeatability across runs
  • Validation before generated code is executed
  • Standard Python packaging and release workflows

Provider Support And Runtime Design

I wanted the library to be useful both for local experimentation and for hosted model APIs. PyFuncAI currently supports local models through Ollama, and remote providers through OpenAI and Gemini.

The runtime is intentionally small: connect to a provider, build a prompt from the requested function behavior and signature, validate the returned source, compile it, and optionally reuse it from cache later. That made it possible to keep the API compact while still supporting a few different workflows.

The Hard Part: Treating Generated Code Seriously

The most important part of the project was not asking a model for Python. It was deciding how much structure to impose on the generated result before allowing it to run.

PyFuncAI validates the generated source, checks that the function signature matches the requested one, restricts imports, and executes the code with a limited builtin environment. It is still experimental, but I wanted the project to acknowledge that generated code is still code, not just text.

Why I Think It Is Interesting

What makes this project interesting to me is that it explores a different interface for writing software. Some functions are still authored manually. Others can be specified declaratively inside the codebase itself and materialized when needed.

I do not think that replaces normal programming. But I do think it opens up a category that is worth exploring seriously: prompt-defined functions as a library abstraction, not just a demo gimmick.
