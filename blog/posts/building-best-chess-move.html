Why I Started This Project

I honestly had been looking for an excuse to code something chess related. I started playing chess seriously just a few years ago, and since then I've become an intermediate level player who occasionally plays in local tournaments.

From the start, I approached it like a real product instead of just a coding exercise. That meant thinking about mobile support, shared hosting constraints, deployment, and a clean user experience, not just the engine logic.

My First Approach: Django + Python-Chess

The original version of BestChessMove was built in Django using python-chess. The goal was to keep the backend responsible for position parsing and move calculation, while the frontend handled the board interaction.

I also wanted the project to work with a cPanel/DirectAdmin style Python app setup, so I created the Django version with that deployment target in mind. Functionally, the app already supported the core workflow: setting positions, syncing FEN, flipping the board, and requesting the next move.

The Biggest Challenge: Hosting Environment, Not Code

The hardest part of this project was not the chess logic. It was shared hosting infrastructure.

The Django app itself validated correctly. WSGI imports worked, migrations worked, and the code was running locally. But on the target host, the Python web app layer kept returning 503 Service Unavailable. After debugging logs, the root cause ended up being the host's LiteSpeed/DirectAdmin Python integration: the server was missing the required lswsgi binaries for multiple Python versions.

In other words, the project code was fine, but the server could not actually launch Python web apps correctly. That was an important lesson: deployment assumptions matter as much as implementation details.

The Pivot: Ship a Simpler Stack

Rather than keep fighting a broken Python runtime on shared hosting, I pivoted to a simpler deployment model: HTML + CSS + JavaScript + PHP. I archived the Django implementation for reference and moved the live version to a PHP-based site that the host could serve reliably.

This let me preserve the user-facing functionality while removing the infrastructure blocker. The project still supports position setup, FEN input, board flipping, mobile interaction, and best-move highlighting.

Current Implementation

The current version uses a JavaScript chessboard UI and a PHP API endpoint for move calculation. The move analysis is handled through the Lichess Cloud Eval API, which gave me a practical way to provide strong move suggestions without running a chess engine process on the shared host.

The core implementation focuses on:

  • Drag-and-drop board editing with FEN synchronization
  • A PHP proxy endpoint for Lichess cloud evaluation
  • Move highlighting and clear result feedback

I also spent time on the UX details: drag-and-drop board editing, mobile touch support, FEN synchronization, and a cleaner interface that works on both desktop and mobile. That part mattered to me because a chess tool only feels useful if entering positions is fast and frictionless.

What I Took Away From It

This project reinforced something I keep seeing in real-world software work: the "best" architecture on paper is not always the best architecture for the environment you actually have.

I still like the Django version and I'm glad I built it, because it helped me validate the core design and backend logic. But the successful version was the one that adapted to deployment reality and shipped. (You can still find it under _django in the repo!)

And honestly, that made the project more satisfying. I got to build something around a game I love, solve real integration problems, and end up with a tool that people can actually use.
